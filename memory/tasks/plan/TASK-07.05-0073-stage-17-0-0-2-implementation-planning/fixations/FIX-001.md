## Accepted Decisions

### Cardinality declaration model

- `cardinality` додається і в `provides`, і в `requires`.
- Default cardinality: `single`.
- `requires.kind` не використовується для single/multi semantics. `kind` лишається
  dependency kind: `external | shared`.
- Multi dependency semantics:
  - `required: true` означає, що потрібен мінімум один contributor/provider;
  - `required: false` дозволяє відсутність contributors;
  - `getAll(token)` для optional missing multi dependency повертає `[]`.

### Сукупна declaration model

Validation має враховувати не тільки `provides`, а сукупну declaration/registration model:

- `provides`;
- `requires`;
- composer bindings;
- module setup registrations.

Правила:

- single capability: один token може бути provided тільки одним module;
- duplicate single capability produces `SAGIFIRE_IOC_DUPLICATE_SINGLE_CAPABILITY`;
- multi capability: один token може бути provided багатьма modules, кожен module є
  contributor;
- single/multi conflict for same token produces
  `SAGIFIRE_IOC_CAPABILITY_CARDINALITY_CONFLICT`;
- token не може бути single в одному місці й multi в іншому;
- declared multi capability має реєструватися через `add()`, а не `bind()`;
- declared single capability має реєструватися через `bind()`, а не `add()`;
- registration mismatch отримує окремий diagnostic у namespace `SAGIFIRE_IOC_*`.

### Runtime gating

Composed runtime має enforce public surface rule:

- `get()` для multi token має падати;
- `getAll()` для single token має падати;
- можливі diagnostic codes:
  - `SAGIFIRE_IOC_GET_USED_FOR_MULTI_TOKEN`;
  - `SAGIFIRE_IOC_GET_ALL_USED_FOR_SINGLE_TOKEN`.

Це не тільки container behavior, а public composed runtime rule.

### Ordering

Multi-provider resolution order deterministic:

1. effective composer module registration order;
2. registration order inside each module `setup()`;
3. composition-root additions after module contributions, if `composer.add()` is supported;
4. composition-root additions ordered by registration order.

### Inspection model

Multi-capability graph має показувати cardinality and providers. Target shape:

```ts
interface GraphCapabilityNode {
    readonly tokenId: string
    readonly kind: string
    readonly cardinality: 'single' | 'multi'
    readonly providers: readonly GraphCapabilityProvider[]
}

interface GraphCapabilityProvider {
    readonly moduleId: string
    readonly registrationKind: 'bind' | 'add'
    readonly registrationIndex: number
}
```

Composition-root additions may require a provider identity such as `moduleId: undefined` or
explicit composition-root provider metadata; concrete shape is decided during task
implementation.

### Graph-aware adapter API

`using()` не отримує `{ get }` або equivalent resolver context.

Acceptable shape:

```ts
composer
    .adapt(CONTACT_REQUESTS_AUTH_READER)
    .from(AUTH_PUBLIC_API)
    .using((auth) => ({
        getCurrentUser: () => auth.getCurrentUser()
    }))
```

Rejected shape for graph-aware adapter API:

```ts
composer
    .adapt(CONTACT_REQUESTS_AUTH_READER)
    .from(AUTH_PUBLIC_API)
    .using((auth, { get }) => {
        const permissions = get(PERMISSIONS_PUBLIC_API)

        return ...
    })
```

Reason: second form makes graph incomplete again.

Adapter validation rules:

- target token має бути required port / external dependency token;
- source token має бути public capability or explicit valid composition-level source;
- source token має бути single capability in first adapter slice;
- source token має бути provided або explicitly bound in composition;
- factory не виконується під час validation;
- graph має показувати adapter edge.

Diagnostics:

- `SAGIFIRE_IOC_ADAPTER_SOURCE_MISSING`;
- `SAGIFIRE_IOC_ADAPTER_SOURCE_PRIVATE`;
- `SAGIFIRE_IOC_ADAPTER_SOURCE_CARDINALITY_MISMATCH`;
- `SAGIFIRE_IOC_ADAPTER_TARGET_INVALID`.

### Adapter-aware cycles

First adapter slice:

- adapter graph visibility: yes;
- adapter source validation: yes;
- adapter-aware cycle detection: no.

Before `0.0.2` release:

- adapter-aware cycle detection must be implemented fully.

### Child scope cache inheritance

Child scope model:

- child scope inherits parent scope values;
- child scope can override values;
- child scope has its own scoped provider cache;
- child scope does not reuse parent scoped provider instances by default.

Reason: transaction/preview/impersonation overlays often override `DB_SESSION`,
`TRANSACTION_CONTEXT`, `CURRENT_USER`, `TENANT_ID`, `LOCALE` or `PREVIEW_MODE`. Reusing
parent scoped instances can accidentally keep services that already captured old scope
values.
