## Stage 8 Implementation Plan

### TASK-06.29-0016: Diagnostics error foundation

Purpose:

- implement `SagifireIocError` in `packages/ioc/src/diagnostics.ts`;

  `cause`;
- add type guard such as `isSagifireIocError(error)`;
- migrate existing Stage 3-7 public typed errors to extend `SagifireIocError`;
- preserve existing public class names and code strings;

### TASK-06.29-0017: Diagnostic reports and formatting

Purpose:

- implement `DiagnosticSeverity`;

- add a small error-to-diagnostic conversion helper if needed by the public API;
- make output deterministic, readable and useful for humans and Codex;
- support readable rendering of details from Stage 3-7 typed errors;
- export reports/formatter through `@sagifire/ioc/diagnostics` and root where appropriate.

## Stage 8 Decisions

- Error code naming convention is `SAGIFIRE_IOC_<AREA>_<REASON>` in stable uppercase snake
  case.
- Existing Stage 3-7 code strings are preserved unless the implementation finds a direct
  conflict; any deviation must be documented in the run result.
- `code` is public API. Implementation may add new codes, but should not silently rename
  existing codes.
- `details` should be structured data made from token IDs, actions, expected/actual kinds,
  lifecycle modes, scope reasons and cycle paths. Details must not expose provider values
  or private runtime internals.
- `cause` should preserve underlying error cause when the platform/toolchain allows it.
- `formatDiagnostics()` must not depend on Node-only APIs, terminal colors or process
  globals.
- Unknown non-IoC errors may be represented as a generic diagnostic, but should not be
  falsely classified as a specific IoC failure.
- Stage 8 does not implement composer/module graph diagnostics. Module IDs, capabilities,
  required ports, bindings and module cycle diagnostics start with Stage 9+ composer work.
