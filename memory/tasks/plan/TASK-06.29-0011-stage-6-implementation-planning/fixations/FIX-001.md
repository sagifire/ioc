## Stage 6 Implementation Plan

Stage 6 implementation task має змінювати тільки core package scope/container surface:

- `packages/ioc/src/container.ts`
- `packages/ioc/src/context.ts`
- `packages/ioc/src/index.ts`, якщо потрібен root re-export
- tests для scope runtime behavior і public type inference
- package export smoke tests, якщо потрібно оновити expectations для
  `@sagifire/ioc/context`
- docs/README тільки там, де Stage 6 public API description інакше буде misleading

Planned public API baseline:

- `runtime.createScope(options?)`.
- `runtime.withScope(callback)` і `runtime.withScope(options, callback)`.
- `Scope.get(token)`.
- `Scope.tryGet(token)`.
- `Scope.getAll(token)`.
- `Scope.dispose(): Promise<void>`.
- `CreateScopeOptions` або equivalent explicit structure for scope-local single values and
  scope-local multi values.
- `.scoped()` lifetime modifier on `LifetimeBinding`.
- `InvalidScopeError` або equivalent minimal typed error.

Planned Stage 6 decisions:

- `ProviderLifetime` розширюється до `singleton | transient | scoped`.
- `.scoped()` applies to sync single-provider `toFactory()` / `toClass()` and
  multi-provider `toFactory()` contributions.
- `toValue()` лишається singleton by definition і не отримує `.scoped()`.
- Scoped provider має one instance/value per scope.
- Runtime-level `get()` / `tryGet()` / `getAll()` fail with `InvalidScopeError` або
  equivalent, якщо для resolution потрібен scoped provider and no active scope exists.
- Scope-level `get()` / `tryGet()` / `getAll()` resolve through the same runtime provider
  registry but carry active scope identity and scoped cache.
- Factory `ResolutionContext` is scope-bound when provider is resolved through `Scope`.
  Scoped dependencies and scope-local values are visible inside that factory only through
  the active scope resolution path.
- Scope-local values are supplied at scope creation time and are not mutable through public
  scope APIs after scope creation.
- Scope-local single values override runtime single-provider resolution for the same token
  ID within that scope.
- Scope-local multi values extend runtime multi-provider collections in runtime-first,
  scope-local-after order.
- Missing token with local single value behaves as scope-local single provider:
  `scope.get()` returns the value, `scope.tryGet()` returns the value, `scope.getAll()`
  fails as single/multi mismatch.
- Missing token with local multi values behaves as scope-local multi provider:
  `scope.getAll()` returns local values, `scope.get()` / `scope.tryGet()` fail as
  single/multi mismatch.
- Runtime single registration plus local multi values for same token ID fails.
- Runtime multi registration plus local single value for same token ID fails.
- Local single and local multi entries for same token ID fail.
- Duplicate local single values for same token ID fail.
- Multiple local multi values for same token ID are allowed and preserve declaration order.
- `scope.dispose()` is idempotent.
- After scope disposal, `scope.get()` / `scope.tryGet()` / `scope.getAll()` fail with
  readable typed error.
- `runtime.withScope()` disposes scope in `finally`, including callback throw/reject cases,
  and returns/rejects with the callback result.
- Stage 6 keeps `get()` sync and does not add async provider/resource behavior.

Planned minimal errors:

- `InvalidScopeError` або equivalent for:
  - resolving scoped provider without active scope;
  - resolving from disposed scope;
  - invalid scope-local single/multi conflict.
- Reuse `ProviderKindMismatchError` where the failure is specifically single/multi token
  kind mismatch.
- Reuse `ProviderCycleError` for cycles through scope-bound `get()` / `getAll()`.

Ці errors можуть мати stable local `code` strings і readable messages, але Stage 6 не має
вводити `SagifireIocError`, `DiagnosticReport` або diagnostic formatting.

## Rationale

Stage 6 має розширити існуючий immutable runtime, а не створити другий mutable container.
Тому scope-local values задаються під час створення scope і не додаються/замінюються
публічним API після цього. Це зберігає predictable lifecycle і не заохочує service locator
usage в application layer.

Precedence model розділяє single і multi contracts. Override дозволений тільки для
single-value dependency, де scope дійсно несе request/operation-specific context. Extend
дозволений тільки для collection/contribution dependency. Якщо змішати ці контракти для
одного token ID, бібліотека має fail fast, бо silent conversion приховує dependency graph.

Одна implementation task є кращою за штучний split, бо scoped lifetime, local values,
disposed-scope checks і `withScope()` спільно визначають active-scope resolution path. Split
ризикував би залишити проміжний API, де scope існує без повної resolution semantics.
