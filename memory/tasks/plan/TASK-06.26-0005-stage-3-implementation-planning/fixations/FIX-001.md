## Stage 3 Implementation Plan

Stage 3 implementation task має змінювати тільки core package token surface:

- `packages/ioc/src/tokens.ts`
- `packages/ioc/src/index.ts`, якщо потрібен root re-export
- tests для runtime behavior і type inference
- docs/comments тільки там, де вони прямо потрібні для public API clarity

Planned API:

- `Token<TValue>` interface з `id`, optional `description` і phantom `__type`.
- `token<TValue>(id, options?)`.
- `namespace(id)` helper, який створює stable prefixed token IDs через `.`.
- `InvalidTokenIdError` як мінімальна token-specific validation error з readable message
  і stable `code`, без `SagifireIocError`, `DiagnosticReport` або formatter.

Planned validation baseline:

- ID має бути non-empty string.
- ID не має містити leading/trailing whitespace, whitespace всередині або control chars.
- ID складається з ASCII letters/digits і символів `.`, `-`, `_`, `:`, `/`.
- Namespace prefix і local token ID валідяться окремо перед join.

## Rationale

Tokens є фундаментом для всіх наступних stages. Якщо Stage 3 потягне container або
composer behavior, наступні етапи втратять чисту межу acceptance criteria. Якщо ж token
validation відкласти повністю, container stage будуватиметься на нечіткому identity
contract. Тому Stage 3 має реалізувати вузький, перевірений token identity layer і
мінімальну локальну validation error, залишивши повний diagnostics framework для Stage 8.

Vitest `expectTypeOf` достатній для Stage 3, бо dependency вже присутня, inference
перевіряється поруч із runtime tests, а окремий type-test package можна буде додати
пізніше тільки за конкретної потреби.
