## Stage 15 Implementation Plan

Stage 15 split into seven sequential implementation tasks.

### TASK-07.02-0055: Repository governance artifacts

Purpose:

- replace placeholder root/package license files with Apache License 2.0 text;
- update root/package NOTICE content;
- add `CONTRIBUTING.md`, `SECURITY.md` and `TRADEMARKS.md`;
- record GitHub Issues as the primary non-sensitive contact/support channel;
- document `@sagifire/ioc` as a product mark without claiming trademark registration;
- update package manifests from `UNLICENSED` to `Apache-2.0`.

### TASK-07.02-0056: Package publish metadata

Purpose:

- add publish-ready package metadata for all publishable packages;
- set repository/homepage/bugs metadata with GitHub Issues as the issue URL;
- configure public package access and package file contents intentionally;
- preserve root workspace privacy and package boundary rules.

### TASK-07.02-0057: Changesets versioning and changelog

Purpose:

- add Changesets or document a concrete equivalent if Changesets is blocked;
- configure package versioning strategy for the three publishable packages;
- add version/changelog scripts and baseline release notes workflow;
- avoid actual publishing in this task.

### TASK-07.02-0058: GitHub Actions CI quality gates

Purpose:

- add CI workflow for install, build, typecheck, tests and lint;
- keep CI separate from publishing;
- verify workflow YAML and local commands where practical.

### TASK-07.02-0059: Package dry-run validation

Purpose:

- add npm package dry-run validation for publishable packages;
- verify tarball/package contents include `dist`, README, LICENSE, NOTICE and package
  metadata;
- verify package exports work from packed artifacts where practical.

### TASK-07.02-0060: npm publish workflow and provenance

Purpose:

- add release/publish workflow using the chosen versioning flow;
- support npm provenance where practical;
- use repository secrets and GitHub OIDC permissions without committing credentials;
- avoid actual publish unless explicitly approved by the human.

### TASK-07.02-0061: Release docs and final hardening

Purpose:

- update README/package docs from "not released yet" wording to the accepted release
  governance model where implementation made that true;
- link license, contributing, security, trademarks and changelog documents;
- run final release/readiness consistency checks;
- sync Project Memory after Stage 15 completion.

## Stage 15 Decisions

- Use Apache License 2.0 for repository and publishable packages.
- Treat `@sagifire/ioc` as a product mark and document usage rules.
- Use GitHub Issues as the primary project support/contact channel.
- Governance artifacts task must be first in Stage 15.
- Changesets is the planned default for versioning/changelog in the pnpm monorepo; an
  implementation task may document and justify an equivalent replacement if a concrete
  blocker appears.
- Release automation should support npm provenance where practical.
- Actual npm publishing requires explicit human approval and valid external repository/npm
  credentials; Stage 15 repository work must not invent or embed credentials.
- Release tasks must preserve all core/package boundaries and must not change runtime API
  behavior unless a separate task-level need is documented.
