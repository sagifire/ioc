## Stage 9 Implementation Plan

Stage 9 split into five sequential implementation tasks.

### TASK-06.30-0019: Module definition foundation

Purpose:

- replace composer placeholder with object-configuration module API foundation;
- implement `defineModule()`;
- define `ModuleDefinition`, `ModuleDependencyDefinition`, `ModuleCapabilityDefinition`,
  `ModuleSetupFunction`, `ModuleSetupContext` and `ModuleSetupResult`;
- normalize dependency defaults: `required = true`, `kind = 'external'`;
- validate/freeze module definition inputs where useful;
- export module definition API through `@sagifire/ioc/composer` and root where
  appropriate.

### TASK-06.30-0020: Composer builder, bindings and static validation

Purpose:

- implement `createComposer()`, `composer.use()`, `composer.bind()` and
  `composer.validate()`;
- store composition-level bindings without executing module setup;
- validate duplicate module IDs, missing required ports, duplicate declared/provided
  single capabilities where statically knowable and invalid binding targets;
- return `DiagnosticReport` from validation;
- add Stage 9 composer typed errors for validation.

### TASK-06.30-0021: Module setup context and private providers

Purpose:

- execute module setup during composition;
- support module setup `bind()` / `add()` for private and exported providers;
- provide module-bound resolution context for setup/provider factories;
- enforce that module internals are not resolved by other modules;
- keep module setup unable to access mutable composer after composition starts.

### TASK-06.30-0022: Composed runtime and capability registry

Purpose:

- implement `composer.compose()`;
- build the internal container from module providers and composer bindings;
- return immutable composed runtime wrapper;
- expose only declared exported capabilities through public runtime resolution APIs;
- allow bindings to satisfy required ports without automatically becoming public
  capabilities;
- preserve container scope, async and disposal behavior through the composed runtime.

### TASK-06.30-0023: Inspection API

Purpose:

- implement `composer.inspect()`, `composer.getGraph()` and `runtime.inspect()`;
- expose safe metadata for registered modules, capabilities, required ports, bindings,
  provider kinds/lifetimes and validation status;
- avoid exposing provider values, resource instances, scope-local values or private
  runtime internals;
- keep module cycle detection and dependency-edge cycle analysis out of Stage 9.

## Stage 9 Decisions

- Stage 9 uses explicit object configuration first. DSL helpers remain Stage 11.
- `defineModule()` returns a normalized/frozen module definition or an equivalent immutable
  value suitable for safe reuse.
- Required port defaults are `required = true` and `kind = 'external'`.
- Composition bindings satisfy required ports but do not automatically become exported
  runtime capabilities.
- Public runtime capability access is based on declared module `provides` metadata and the
  actual provider registrations created during module setup.
- Module private providers are resolvable only by their owning module's setup/provider
  contexts.
- `composer.validate()` returns a `DiagnosticReport`; `composer.compose()` must validate
  and throw a typed `ComposerValidationError` or equivalent if the graph is invalid.
- `composer.validate()` may stay static/synchronous for module definitions and composer
  bindings; compose-time validation may add diagnostics that require setup execution.
- Existing container provider errors remain valid and should be surfaced through Stage 8
  diagnostics foundation instead of wrapped unnecessarily.
- Stage 9 does not implement module cycle detection, capability dependency edges, binding
  dependency edges or cycle path diagnostics. Those belong to Stage 10.
