## Stage 7 Implementation Plan

Stage 7 implementation task має змінювати core package lifecycle/container/context surface:

- `packages/ioc/src/container.ts`
- `packages/ioc/src/context.ts`
- `packages/ioc/src/lifecycle.ts`
- `packages/ioc/src/index.ts`, якщо потрібен root re-export
- tests для async providers, resources, disposal, scopes and public type inference
- package export smoke tests for `@sagifire/ioc/lifecycle`, `./container`, `./context`
  and root exports where needed
- docs/README only where Stage 7 public API description would otherwise be misleading

Planned public API baseline:

- `bind(token).toAsyncFactory(factory)`.
- `bind(token).toAsyncResource(factory)`.
- async binding lifecycle/mode helpers for valid combinations of `singleton()`,
  `transient()`, `scoped()`, `eager()` and `lazy()`.
- `runtime.getAsync(token)`.
- `runtime.tryGetAsync(token)`.
- `runtime.dispose()`.
- `scope.getAsync(token)`.
- `ResolutionContext.getAsync(token)` and `ResolutionContext.tryGetAsync(token)`.
- `Resource<TValue>` або equivalent:
  `value: TValue` and optional `dispose(): void | Promise<void>`.
- Minimal typed errors:
  - `AsyncProviderAccessError`;
  - `RuntimeDisposedError`;
  - `ScopeDisposedError`.

Planned Stage 7 decisions:

- Stage 7 adds async single-provider bindings through `bind()`.
- Stage 7 does not add async multi-provider contributions through `add()`.
- Stage 7 does not add `getAllAsync()` / `scope.getAllAsync()`.
- Sync providers are valid through `getAsync()` / `tryGetAsync()` as resolved Promises.
- Async eager singleton providers/resources initialize during `freeze()`.
- Async eager singleton providers/resources are available through sync `get()` after
  `freeze()` and through `getAsync()`.
- Async lazy providers/resources initialize on `getAsync()`.
- Async lazy providers/resources are never available through `get()` / `tryGet()`, even
  after initialization.
- `get()` / `tryGet()` on async lazy provider/resource throws `AsyncProviderAccessError`
  with token ID and guidance to use `getAsync()` or mark provider eager.
- `toAsyncFactory()` defaults to transient lazy provider.
- Async factory providers may be `transient`, `singleton` or `scoped`.
- Eager async factory initialization is valid only for singleton providers.
- `toAsyncResource()` requires explicit `singleton()` or `scoped()` ownership.
- Async resources default to lazy initialization unless `eager()` is explicitly chosen.
- Eager async resources are valid only for singleton resources.
- Scoped async providers/resources initialize lazily through `scope.getAsync()`.
- Runtime-level resolution of scoped async providers/resources without active scope fails
  with the same invalid-scope model as Stage 6.
- Failed lazy async initialization is not cached by default; a later `getAsync()` retries.
- In-flight lazy initialization is de-duplicated for singleton and scoped lifetimes and
  cleared on rejection.
- Async provider cycles through `get()` / `getAsync()` fail with readable token ID path,
  reusing `ProviderCycleError` or equivalent.
- `runtime.dispose()` disposes initialized singleton resources in reverse initialization
  order where possible and is idempotent.
- `scope.dispose()` disposes initialized scoped resources in reverse initialization order
  where possible and remains idempotent.
- `runtime.dispose()` does not maintain a global registry of live scopes and does not
  silently dispose scopes. Scope disposal remains explicit through `scope.dispose()` and
  `runtime.withScope()`.
- After runtime disposal, runtime resolution and scope creation fail with
  `RuntimeDisposedError`.
- After scope disposal, scope resolution fails with `ScopeDisposedError`.
- Disposer failures should not be hidden. Stage 7 may reject disposal with the first
  disposer failure after attempting best-effort cleanup of remaining initialized resources;
  aggregate diagnostic formatting remains Stage 8+.

Planned minimal errors:

- `AsyncProviderAccessError` for invalid sync access to async lazy provider/resource.
- `RuntimeDisposedError` for runtime resolution or scope creation after runtime disposal.
- `ScopeDisposedError` for scope resolution after scope disposal.
- Reuse `InvalidScopeError` for resolving scoped provider without active scope.
- Reuse `ProviderCycleError` for cycles through async and mixed sync/async resolution.

These errors may have stable local `code` strings and readable messages, but Stage 7 must
not introduce `SagifireIocError`, `DiagnosticReport` or diagnostic formatting.

## Rationale

Stage 7 should be one implementation task because async provider access, initialization
mode, value/resource caching, retry behavior and disposal are one lifecycle model. Splitting
them would create a risky intermediate state where async values can be initialized without
clear ownership or cleanup semantics.

Async multi-provider support is intentionally out of scope. The canonical runtime API and
historical Stage 7 source define `getAsync()` / `tryGetAsync()` for single-provider
resolution, but do not define `getAllAsync()`. Adding async collection resolution now would
invent a larger API contract than the current roadmap requires.

Runtime disposal should not create a hidden global scope registry. Scopes are explicit
lifecycle boundaries. `withScope()` and `scope.dispose()` own scoped resources; runtime
disposal owns initialized singleton resources and prevents further runtime/scope
resolution.
