## Stage 11 Implementation Plan

Stage 11 split into four sequential implementation tasks.

### TASK-07.01-0029: Module DSL foundation

Purpose:

- implement `module()` DSL as an optional ergonomic layer over `defineModule()`;
- keep the produced module definition compatible with existing composer APIs;
- preserve required port and capability metadata visibility;
- export the initial DSL API from `@sagifire/ioc/dsl` and root if appropriate;
- leave `defineApp()`, `adapt()` and composition-level bind helper DSL for later tasks.

### TASK-07.01-0030: `defineApp()` DSL and object-config conversion

Purpose:

- implement `defineApp()` or equivalent app-level DSL that collects module definitions and
  composition bindings into explicit composer configuration;
- provide a deterministic conversion path to `createComposer()`, `composer.use()` and
  `composer.bind()`;
- expose validation/inspection/compose behavior through existing composer semantics;
- avoid global app registries and hidden framework assumptions.

### TASK-07.01-0031: Bind helper DSL and `adapt()`

Purpose:

- implement ergonomic bind helper declarations for composition-level bindings;
- implement `adapt()` for explicit required-port adapters over public capabilities or
  context-accessible dependencies;
- keep adapter dependencies explicit in code and visible through required-port/binding
  graph metadata;
- preserve no-hidden-factory-execution validation boundary.

### TASK-07.01-0032: DSL hardening, exports and docs

Purpose:

- harden DSL type inference and runtime conversion regressions;
- verify root and `@sagifire/ioc/dsl` package exports;
- ensure DSL-generated configuration has inspection parity with equivalent object API;
- update README/docs enough to explain the DSL without implying decorators, auto-discovery
  or service locator behavior;
- keep Stage 12 testing helpers and Stage 13 Next.js adapters out of scope.

## Stage 11 Decisions

- DSL is optional and must not be required for any object-configuration workflow.
- `module()` is an ergonomic declaration layer over `defineModule()`, not a new module
  runtime.
- `defineApp()` is an ergonomic application composition layer over `createComposer()` and
  existing composer methods, not a global app registry.
- `adapt()` is an explicit adapter helper for composition bindings; it must not infer
  dependencies by executing user factories during validation.
- DSL declarations must convert to explicit module/composer configuration that remains
  inspectable through existing graph APIs.
- DSL must not hide required ports, capabilities, bindings or dependency edges.
- DSL must not add decorators, `reflect-metadata`, constructor metadata, filesystem
  auto-discovery, framework imports or global mutable containers.
- Stage 11 does not implement `@sagifire/ioc-testing` helpers, graph assertion helpers,
  Next.js adapters or release automation.
