## Stage 5 Implementation Plan

Stage 5 implementation task має змінювати тільки core package container surface:

- `packages/ioc/src/container.ts`
- `packages/ioc/src/index.ts`, якщо потрібен root re-export
- tests для multi-provider runtime behavior і public type inference
- package export smoke tests, якщо потрібно оновити expectations для
  `@sagifire/ioc/container`
- docs/README тільки там, де Stage 5 public API description інакше буде misleading

Planned public API baseline:

- `ContainerBuilder.add(token)`.
- `MultiBindingBuilder` або еквівалентний builder для `add()`.
- `add(token).toValue(value)`.
- `add(token).toFactory(factory)`.
- `.singleton()` і `.transient()` lifetime modifiers для multi-provider factory
  providers.
- `runtime.getAll(token)`.
- `ResolutionContext.getAll(token)` для sync factories.

Planned Stage 5 decisions:

- `bind(token)` лишається single-provider registration API.
- `add(token)` є multi-provider registration API.
- Для одного token ID не можна змішувати `bind()` і `add()`.
- `add()` можна викликати багато разів для одного token ID; order resolution відповідає
  registration order.
- `get(token)` fails для multi-provider token навіть якщо зареєстрований лише один
  multi-provider.
- `getAll(token)` fails для token, зареєстрованого через `bind()`.
- `getAll(token)` повертає empty array для повністю missing token.
- `getAll(token)` повертає fresh mutable `TValue[]` у registration order; mutation цього
  array не мутує runtime state.
- `add().toValue()` є singleton by definition.
- `add().toFactory()` transient by default і підтримує `.singleton()` / `.transient()`.
- Provider cycle detection має працювати через `get()` і `getAll()` resolution paths.
- Token ID лишається provider key; token object identity не використовується для matching.

Planned minimal errors:

- Reuse existing `DuplicateProviderError` або equivalent для duplicate `bind()`.
- `ProviderKindMismatchError`, `MultiProviderResolutionError` або equivalent для
  incompatible single/multi registration або resolution.
- Reuse existing `ProviderCycleError` або equivalent для cycles через `get()` / `getAll()`.

Ці errors можуть мати stable local `code` strings і readable messages, але Stage 5 не має
вводити `SagifireIocError`, `DiagnosticReport` або diagnostic formatting.

## Rationale

Stage 5 є природним розширенням Stage 4 provider registry, а не окремим module/composer
шаром. Найбільший ризик - зробити `getAll()` занадто permissive і стерти межу між
single-provider contract та collection/contribution contract. Strict model зберігає
архітектурну ясність: якщо dependency є collection, її реєструють через `add()` і читають
через `getAll()`.

Публічний тип `TValue[]` для `getAll()` лишається сумісним із canonical architecture і
JavaScript-friendly ergonomics. Runtime immutability забезпечується не readonly типом, а
fresh array на кожен `getAll()` call і відсутністю exposure internal provider storage.
