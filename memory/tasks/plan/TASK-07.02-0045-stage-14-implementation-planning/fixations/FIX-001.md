## Stage 14 Implementation Plan

Stage 14 split into eight sequential implementation tasks.

### TASK-07.02-0046: README and package documentation

Purpose:

- replace stage-tracking root README copy with a product-oriented overview and quickstart;
- update package READMEs for `@sagifire/ioc`, `@sagifire/ioc-next` and
  `@sagifire/ioc-testing`;
- create or update docs navigation if needed;
- keep all examples aligned with implemented public API only.

### TASK-07.02-0047: Core architecture, container and async docs

Purpose:

- document architecture, package boundaries and reference composition model;
- expand container docs for tokens, providers, lifetimes, scopes and sync resolution;
- expand async model docs for eager/lazy async providers, async resources and disposal;
- include focused snippets that are small enough to keep readable.

### TASK-07.02-0048: Composer, modules and diagnostics docs

Purpose:

- document module definitions, required ports, capabilities, bindings and module
  isolation;
- document composer validation, inspection, dependency edges and module cycle diagnostics;
- document diagnostics types, formatter and testing diagnostic assertions;
- explain object API first and DSL as optional parity layer.

### TASK-07.02-0049: Testing and Next integration docs

Purpose:

- expand testing docs for isolated runtimes, overrides, test composers, fake modules,
  module harnesses and assertions;
- expand Next integration docs for cached runtime, request context, route scopes and server
  action scopes;
- keep framework boundary examples thin and call module public APIs.

### TASK-07.02-0050: `basic-node` and `module-composition` examples

Purpose:

- add `examples/basic-node` with minimal runtime/container usage;
- add `examples/module-composition` with modules, required ports, binding adapter,
  validation/inspection and runtime resolution;
- ensure examples are executable or at least typechecked through existing workspace
  scripts where practical.

### TASK-07.02-0051: `async-db-resource` and `testing-overrides` examples

Purpose:

- add `examples/async-db-resource` demonstrating async resource initialization,
  `getAsync()` and disposal;
- add `examples/testing-overrides` demonstrating test runtime/composer overrides, fake
  modules, module harnesses and graph/diagnostic assertions;
- verify examples do not mutate frozen runtimes.

### TASK-07.02-0052: Next App Router example hardening

Purpose:

- harden existing `examples/next-app-router` from narrow skeleton to Stage 14 example;
- improve README and file organization around runtime cache, request context, route
  handler and server action boundaries;
- avoid adding a full Next.js dependency unless the task documents why it is necessary and
  asks for permission.

### TASK-07.02-0053: Migration guide and final docs hardening

Purpose:

- write migration guidance from common DI containers to typed tokens, explicit providers,
  modules, required ports and bindings;
- run final documentation/example consistency checks;
- verify no docs claim unimplemented Stage 15 release automation;
- sync memory after Stage 14 completion.

## Stage 14 Decisions

- Stage 14 is docs/examples only and must not introduce new runtime behavior.
- Public documentation should describe implemented API, not aspirational future API.
- Object-configuration API remains first-class in docs; DSL may be shown as optional
  ergonomic layer only after explicit object/API concepts are clear.
- Examples must keep dependency graph explicit and inspectable.
- Examples must not use decorators, `reflect-metadata`, filesystem auto-discovery, route
  scanning, global mutable containers or hidden service locator access.
- Next examples must keep business logic behind module public APIs and use adapter helpers
  only at framework boundaries.
- Testing examples must use `@sagifire/ioc-testing` helpers before composition/freeze and
  must not mutate frozen runtimes.
- If a docs/example task discovers a public API gap, it should record a follow-up instead
  of silently adding behavior outside Stage 14.
- Stage 14 does not implement release automation, changesets, publishing workflow or CI
  release jobs. Those remain Stage 15 scope.
