## Stage 10 Implementation Plan

Stage 10 split into three sequential implementation tasks.

### TASK-06.30-0025: Dependency edge model

Purpose:

- extend module graph public types with dependency edge metadata;
- infer capability dependency edges from required ports satisfied by module capabilities;
- infer binding dependency edges from required ports satisfied by explicit composer
  bindings;
- keep edge ordering deterministic and safe for inspection;
- avoid executing setup/provider/binding factories to infer hidden dependencies;
- leave module cycle validation for the next task.

### TASK-06.30-0026: Module cycle diagnostics

Purpose:

- detect module-level cycles over capability dependency edges;
- add typed cycle diagnostics with module ID path and token/capability path;
- integrate cycle diagnostics into `composer.validate()`, `composer.inspect().validation`,
  `composer.prepare()` and `composer.compose()`;
- keep valid acyclic graphs composing successfully.

### TASK-06.30-0027: Runtime inspection and binding-edge hardening

Purpose:

- make composed `runtime.inspect()` expose the final Stage 10 graph shape for successful
  acyclic graphs;
- harden binding-edge semantics, including graphs where explicit bindings intentionally
  avoid module-to-module cycles;
- replace Stage 9 guard assertions that expected no edges/cycles with Stage 10 assertions;
- update package export smoke tests and minimal docs if public API text would otherwise be
  misleading.

## Stage 10 Decisions

- Stage 10 graph edges are explicit inspection metadata, not hidden provider execution.
- Capability dependency edge means a consumer module required port is satisfied by another
  module's declared provided capability.
- Binding dependency edge means a consumer module required port is satisfied by an explicit
  composition-level binding.
- If a required port is satisfied by an explicit binding, the graph records a binding edge
  and does not create a module-to-module capability edge for that required port.
- Module cycles are detected over module-to-module capability dependency edges.
- Binding edges represent composition adapters and do not create module-level cycles by
  themselves.
- `composer.validate()` must not run binding factories, module provider factories or async
  resources to infer dependencies.
- Provider-level cycles discovered inside factories remain existing container/runtime
  diagnostics.
- Cycle diagnostics must include safe structured details: module ID path and token/capability
  path. They must not expose provider values, resource instances, scope-local values or
  private runtime internals.
- Stage 10 does not implement DSL helpers, testing helpers, graph assertion helpers or
  Next.js adapters.
