## Rationale

Stage 2 створює foundation, на який спиратимуться всі наступні stages. Якщо не відділити
планування від реалізації, implementation run може або недобрати package/build plumbing,
або завчасно почати runtime behavior. Окрема implementation task із готовим `RUN-001`
зменшує неоднозначність для наступного агента і тримає Stage 2 в межах roadmap.

`tsup` обрано як planned tool, бо Stage 2 потребує ESM-first output, declarations,
subpath-export-friendly package builds і незалежних package build scripts. Якщо під час
implementation run з'явиться конкретний blocker, заміну build tool треба зафіксувати в
task result і technical memory sync, а не робити приховано.
